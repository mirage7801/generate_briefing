# AI+硬件协同设计愿景自动化简报生成器

## 一、项目简介

本项目是一个基于 Python 和大语言模型的论文简报自动生成工具。

用户输入一篇 PDF 或 TXT 格式的技术论文后，程序可以自动完成：

1. 读取并清洗论文文本；
2. 将长论文切分为多个适合大模型处理的文本块；
3. 从每个文本块中提取专业术语、关键关系、数字和发展趋势；
4. 合并不同文本块中的信息；
5. 生成结构化 JSON 数据；
6. 根据 JSON 自动生成一份三页 PowerPoint 简报。

最终生成的 PPT 包括：

- 第 1 页：论文标题与核心愿景；
- 第 2 页：Hardware、Algorithm、Application 三个抽象层级；
- 第 3 页：近期 2—5 年与远期 6—10 年技术发展时间轴。

整个项目形成了一条从“原始论文”到“结构化内容”，再到“可视化简报”的自动化处理流程。

---

## 二、项目整体工作流程

```text
输入 PDF/TXT
    ↓
提取论文文字
    ↓
清洗和规范文本
    ↓
将论文切分为多个 Chunk
    ↓
逐个 Chunk 提取 NIL
    ↓
合并全部 NIL
    ↓
选择与主题最相关的原文
    ↓
生成最终 JSON
    ↓
检查并修复 JSON
    ↓
自动生成三页 PPT
```

项目没有直接让大模型一次性阅读整篇论文，而是先对论文进行分块，再逐步提取和汇总信息。

这样可以降低长文本超过模型处理范围的风险，也能让每一个阶段的任务更加明确。

# 四、使用的方法、实现的功能与规避的问题

## 4.1 支持 PDF 和 TXT 文档读取

程序通过 `pypdf` 读取 PDF，通过 Python 文件接口读取 TXT。

对于 PDF，程序会逐页提取文字，并将各页内容合并。

对于 TXT，程序会依次尝试：

```text
UTF-8
GBK
```

这两种编码。

### 通俗解释：字符编码

字符编码是计算机保存文字时使用的规则。

UTF-8 是目前较常见的通用编码，GBK 常见于部分中文 Windows 文件。

同一份中文文本使用了不同编码，读取时选错编码就可能出现乱码。

### 实现效果

程序能够处理：

- 英文 PDF；
- 可复制文字的中文或英文 PDF；
- UTF-8 编码的 TXT；
- GBK 编码的中文 TXT。

### 规避的问题

- 输入文件不存在；
- TXT 中文乱码；
- 单个 PDF 页面解析失败；
- PDF 中没有可提取文字；
- 用户输入不支持的文件格式。

对于解析失败的 PDF 页面，程序会发出警告并跳过，而不是因为一页异常就立即终止整个任务。

如果 PDF 是扫描图片，程序会提示需要先进行 OCR。

### 通俗解释：OCR

OCR 是 Optical Character Recognition 的缩写，即“光学字符识别”。

通俗地说，就是把图片中的文字识别成可以复制、搜索和处理的普通文本。

---

## 4.2 对论文文本进行清洗

PDF 提取出的文字通常会包含：

- 多余空格；
- 不统一的换行；
- 页内断行；
- 空白段落；
- 被拆开的普通句子。

程序通过 `clean_text()` 对这些内容进行处理，包括：

1. 统一不同系统的换行符；
2. 合并多余空格；
3. 清除行首和行尾空白；
4. 尽量保留章节和段落边界；
5. 将普通正文中的连续断行重新连接。

### 通俗解释：文本清洗

文本清洗就是在正式分析前，把杂乱的原始文字整理成更统一、更容易处理的形式。

它类似于把一份排版混乱的复印资料重新整理成段落清楚的电子文档。

### 规避的问题

- PDF 每一行都被误认为一个独立段落；
- 多余换行影响模型理解；
- 标题与正文混在一起；
- 不同操作系统的换行方式不一致。

---

## 4.3 自动识别章节标题

程序通过 `detect_section_title()` 判断某个段落是否可能是章节标题。

识别方式包括：

- 数字编号标题，例如 `3.2 Hardware`；
- 罗马数字标题，例如 `IV. Discussion`；
- 常见论文标题，例如 `Abstract`、`Introduction`、`Conclusion`；
- 字数较少且大写字母比例较高的英文短语。

识别出的章节标题会继续传递给后面的文本块。

### 实现效果

每个文本块不仅保存正文，还可以保存它所属的章节名称：

```json
{
  "chunk_id": 3,
  "section_title": "3.2 Hardware",
  "text": "该部分正文"
}
```

### 规避的问题

- 文本被切分后失去章节背景；
- 模型不知道当前内容属于硬件、算法还是应用部分；
- 后续无法判断信息来自论文哪个章节。

---

## 4.4 使用 Chunking 处理长论文

程序不会将整篇论文一次性提交给模型，而是使用 `chunk_text()` 将论文切成多个文本块。

### 通俗解释：Chunking

Chunking 就是“文本分块”。

通俗地说，就是把一篇很长的论文拆成多个较短的阅读单元，让模型一段一段处理。

每个文本块包含：

```json
{
  "chunk_id": 1,
  "section_title": "Introduction",
  "text": "文本块正文"
}
```

程序优先按照以下边界切分：

1. 章节；
2. 段落；
3. 句号、分号和问号；
4. 英文单词之间的空格；
5. 必要时才按字符位置切分。

### 实现效果

长论文可以被拆成若干次模型请求，不需要一次性塞入模型。

用户还可以通过命令行调整：

```text
--max-chars
--overlap
```

其中：

- `max-chars`：一个文本块允许包含的最大字符数；
- `overlap`：相邻文本块之间保留的重复字符数。

### 通俗解释：Overlap

Overlap 是“重叠区”。

例如，上一段最后的 250 个字符会再次出现在下一段开头。

这样可以避免一段重要内容刚好被切分点截开，导致下一段完全缺少前文背景。

### 规避的问题

- 整篇论文超过大模型的上下文长度；
- 句子从中间被切开；
- 前后文本块之间完全失去联系；
- 章节信息在切分后丢失；
- 某个特别长的段落无法处理。

---

## 4.5 使用 NIL 提取中间信息

程序不会让模型阅读一个文本块后直接编写 PPT，而是先要求模型生成 NIL 中间结果。

NIL 包括：

- `Named Terms`：重要术语；
- `Important Actions`：重要动作或关系；
- `Literals/Data`：数字、年份、指标和时间范围；
- `Candidate Insights`：可能与最终简报有关的结论。

### 通俗解释：中间表示

中间表示是原始论文和最终简报之间的一层结构化信息。

它类似于：

```text
阅读原文
    ↓
做读书笔记
    ↓
整理笔记
    ↓
撰写最终报告
```

而不是读完一段原文后立即凭印象写最终答案。

### NIL 示例

```json
{
  "chunk_id": 5,
  "section_title": "Hardware",
  "named_terms": [
    {
      "term": "3D Integration",
      "type": "技术术语",
      "context": "通过垂直集成缩短芯片间的数据移动距离"
    }
  ],
  "important_actions": [
    {
      "subject": "3D Integration",
      "action": "reduces",
      "object": "data movement",
      "meaning": "减少数据传输能耗"
    }
  ],
  "literals_data": [
    {
      "value": "1000x",
      "type": "倍数",
      "context": "单位能耗智能能力的提升目标"
    }
  ],
  "candidate_insights": [
    "硬件创新需要与算法和应用共同演进"
  ]
}
```

### 实现效果

模型不仅会提取名词，还会保留：

- 谁对谁产生作用；
- 技术解决了什么问题；
- 某个数字代表什么；
- 某个趋势可能用于 PPT 哪一部分。

### 规避的问题

- 只提取关键词，不理解关键词；
- 忽略数字、年份和倍数；
- 丢失技术之间的因果关系；
- 每个文本块直接写最终总结，导致前后结构不统一。

---

## 4.6 通过 Prompt 约束模型行为

程序为不同任务设计了不同的 Prompt，包括：

- 单个 Chunk 的 NIL 提取 Prompt；
- 多个 NIL 的合并 Prompt；
- 最终简报 JSON 的生成 Prompt；
- JSON 格式修复 Prompt。

### 通俗解释：Prompt Engineering

Prompt Engineering 即“提示词设计”。

它不是简单问模型“帮我总结”，而是明确告诉模型：

- 扮演什么角色；
- 执行什么任务；
- 不能做什么；
- 必须输出哪些字段；
- 应该按照什么格式输出。

例如，NIL Prompt 明确要求：

```text
只输出严格 JSON
不要输出 Markdown
不要编造原文没有的信息
保留英文技术术语
同时提取名词、动作和数据
```

### 实现效果

模型输出具有更稳定的结构，便于后续程序自动读取。

### 规避的问题

- 模型输出大段解释；
- 输出被 Markdown 代码块包裹；
- 模型添加论文中不存在的事实；
- 每次生成的字段名称不同；
- 只有关键词，没有关系和数据。

---

## 4.7 使用语义提取，而不是直接写死答案

程序会特别关注：

- Intelligence per Joule；
- 效率提升目标；
- Hardware；
- Algorithm；
- Application；
- 近期 2—5 年；
- 远期 6—10 年。

但这些内容在 Prompt 中主要用于帮助模型寻找相关语义，而不是直接作为固定答案写入结果。

最终 Prompt 明确规定：

```text
不要把关键词当成答案，必须根据论文语义提取。
```

### 通俗解释：硬编码

硬编码是指将某个答案直接固定写在程序里。

例如，无论输入哪篇论文，程序都强制输出“Photonics”，这就是硬编码。

本项目要求最终内容必须有论文依据，从而避免所有论文都生成相同答案。

### 实现效果

程序能够根据不同论文中的实际表述生成不同内容，而不是只适用于一份固定材料。

---

## 4.8 合并不同文本块中的 NIL

每个 Chunk 完成 NIL 提取后，程序会将所有 NIL 结果交给模型统一合并。

合并任务包括：

1. 删除重复术语；
2. 合并相似动作关系；
3. 合并重复数据；
4. 保留来源 `chunk_id`；
5. 提取高置信度结论。

### 通俗解释：信息去重

信息去重就是将多次出现、含义相同或高度相似的内容合并为一条。

例如多个章节都提到：

```text
减少数据移动可以降低能耗
```

合并后不需要重复保存很多次，但需要记录它来自哪些文本块。

### 来源追踪

合并后的结果会保留：

```json
{
  "source_chunk_ids": [3, 5, 8]
}
```

这表示该结论来自第 3、5、8 个文本块。

### 通俗解释：Provenance

Provenance 可以理解为“来源追踪”。

它用于记录一个结论是从哪些原始内容中提取出来的，避免结果完全脱离原文。

### 规避的问题

- 相同术语在多个章节反复出现；
- 合并后不知道结论来自哪里；
- 重复信息大量占用最终 Prompt；
- 模型只根据单个段落下结论。

---

## 4.9 重新选择最相关的原文

完成 NIL 合并后，程序不会只依赖中间摘要。

`_select_context()` 会重新从原始 Chunk 中选择与最终任务最相关的内容。

程序重点寻找：

```text
efficiency
Intelligence per Joule
1000
Hardware
Algorithm
Application
2-5
6-10
2035
technology trends
```

然后选取得分较高的几个文本块。

### 通俗解释：关键词打分检索

关键词打分检索是指：

1. 预先准备一组重要词语；
2. 检查每个文本块中出现了多少相关词；
3. 出现越多，得分越高；
4. 选择得分最高的文本块。

它类似于在一本书中搜索关键词，再优先阅读命中次数较多的章节。

### 实现效果

最终模型同时获得：

```text
合并后的 NIL
+
与主题最相关的论文原文
```

若两者冲突，Prompt 要求优先采用原文。

### 规避的问题

- 多次摘要后产生信息失真；
- NIL 合并过程中遗漏关键细节；
- 最终模型只看到二手总结；
- 中间结果与论文原文不一致。

---

## 4.10 使用固定 JSON Schema

最终模型必须输出以下结构：

```json
{
  "paper_title": "...",
  "core_vision": {
    "headline": "...",
    "intelligence_per_joule": "...",
    "target_explanation": "..."
  },
  "layers": [
    {
      "name": "Hardware",
      "role": "...",
      "key_features": ["...", "..."],
      "examples": ["...", "..."]
    },
    {
      "name": "Algorithm",
      "role": "...",
      "key_features": ["...", "..."],
      "examples": ["...", "..."]
    },
    {
      "name": "Application",
      "role": "...",
      "key_features": ["...", "..."],
      "examples": ["...", "..."]
    }
  ],
  "timeline": [
    {
      "stage": "近期",
      "range": "2-5 年",
      "trends": ["...", "..."],
      "meaning": "..."
    },
    {
      "stage": "远期",
      "range": "6-10 年",
      "trends": ["...", "..."],
      "meaning": "..."
    }
  ],
  "one_sentence_summary": "..."
}
```

### 通俗解释：JSON

JSON 是一种结构化数据格式。

它类似于一张栏目已经固定好的电子表格，程序可以准确知道：

- 标题存在哪里；
- 三个层级存在哪里；
- 时间轴存在哪里；
- 每个层级有哪些特征。

### 通俗解释：Schema

Schema 是“数据结构规范”。

它规定数据中必须有哪些字段，以及这些字段应当怎样排列。

### 实现效果

最终 JSON 可以直接交给 PPT 生成函数，不需要人工重新整理。

### 规避的问题

- 模型只输出一篇普通文章；
- 每次使用不同字段名；
- 缺少 Hardware、Algorithm 或 Application；
- 缺少近期或远期时间轴；
- PPT 生成器无法找到对应数据。

---

## 4.11 对最终 JSON 进行结构校验

生成最终 JSON 后，程序会检查：

```text
paper_title
core_vision
layers
timeline
one_sentence_summary
```

是否存在。

同时还会检查：

- `layers` 是否为列表；
- 是否至少包含三个层级；
- `timeline` 是否为列表；
- 是否至少包含两个阶段。

### 实现效果

程序不会在数据明显不完整的情况下继续生成 PPT，而是先报告结构错误。

### 规避的问题

- JSON 语法正确，但缺少关键字段；
- 三层架构只生成了两层；
- 时间轴只生成了一个阶段；
- PPT 生成阶段才发现数据不完整。

---

## 4.12 严格解析 JSON

模型返回结果后，程序使用 Python 的 `json.loads()` 进行严格解析。

同时允许去掉模型偶尔附加的：

```text
```json
```

和：

```text
```
```

代码块标记。

### 实现效果

模型即使误加了 Markdown JSON 代码块，程序仍有机会提取其中的 JSON。

### 规避的问题

- JSON 被代码块包裹后无法解析；
- 结果中存在不合法逗号或引号；
- 字符串不是标准 JSON 格式。

---

## 4.13 使用 JSON Repair 修复格式错误

第一次 JSON 解析失败后，程序会将原始输出再次交给模型，并要求模型只修复格式。

Repair Prompt 明确要求：

```text
不要添加新信息
不要删除关键信息
不要输出 Markdown
不要解释
只输出合法 JSON
```

### 通俗解释：JSON Repair

JSON Repair 即“JSON 修复”。

通俗地说，就是让模型充当格式校对员，只修改：

- 括号；
- 引号；
- 逗号；
- JSON 结构。

而不是重新分析论文或重新编写内容。

### 实现效果

轻微格式错误不一定导致整次论文处理作废。

### 规避的问题

- 少一个括号导致程序完全失败；
- 多一个逗号导致全部结果无法使用；
- 为修复格式重新分析整篇论文；
- 修复时模型随意增加新事实。

---

## 4.14 支持 OpenAI-compatible API

程序通过 OpenAI Python SDK 调用大模型，同时允许用户设置：

```text
--api-key
--base-url
--model
```

因此可以连接：

- OpenAI 官方接口；
- DeepSeek；
- 其他兼容 OpenAI Chat Completions 格式的平台。

### 通俗解释：OpenAI-compatible API

OpenAI-compatible API 是指调用方法与 OpenAI 接口相似的模型服务。

程序只需要修改：

```text
API 地址
模型名称
API Key
```

就可以连接不同平台，而不需要重写整套模型调用代码。

### 实现效果

同一份脚本能够切换不同的大模型服务。

### 规避的问题

- 项目只能绑定一个模型平台；
- 更换模型时需要大幅修改代码；
- API 地址和模型名称被固定写死。

---

## 4.15 多种 API Key 配置方式

API Key 的读取顺序包括：

1. `--api-key` 命令行参数；
2. `API_KEY` 环境变量；
3. `OPENAI_API_KEY` 环境变量；
4. `DEEPSEEK_API_KEY` 环境变量；
5. 运行时隐藏输入。

运行时隐藏输入使用 `getpass`，输入内容不会显示在终端。

### 通俗解释：环境变量

环境变量是操作系统为程序保存配置的一种方式。

它可以让程序读取 API Key，而不需要将 Key 直接写进源代码。

### 实现效果

用户可以根据自己的环境选择不同配置方式。

### 规避的问题

- API Key 直接写在 Python 代码中；
- 源代码上传 GitHub 后泄露 Key；
- 输入 Key 时明文显示；
- 项目只能读取某一个固定名称的环境变量。

## 4.16 对错误信息中的 API Key 进行脱敏

模型调用失败时，程序会检查异常信息中是否包含当前 API Key。

如果包含，则替换为：

```text
[REDACTED]
```

### 通俗解释：脱敏

脱敏是指隐藏日志中的敏感信息。

例如将真实 API Key：

```text
sk-xxxxxxxxxxxxxxxx
```

替换为：

```text
[REDACTED]
```

### 规避的问题

- API Key 出现在错误日志；
- 用户截图报错时泄露密钥；
- 调试信息被上传后暴露账号权限。

## 4.17 统一处理程序异常

项目定义了：

```python
class BriefingError(RuntimeError):
```

用于表示可以向用户说明的程序错误。

主程序会分别处理：

- 项目业务错误；
- 用户主动中断；
- 其他异常。

### 通俗解释：异常处理

异常处理就是程序遇到问题时，不直接显示一大段难以理解的内部报错，而是给出更明确的提示。

例如：

```text
Input file does not exist
No usable text could be extracted
API key cannot be empty
Final briefing JSON must contain three layers
```

### 实现效果

错误更容易定位，程序退出方式也更加统一。

### 规避的问题

- 文件错误表现为难懂的 Python 堆栈；
- 用户不知道问题出现在读取、模型还是 PPT 阶段；
- 按下 `Ctrl+C` 后出现大量异常信息。

项目使用 `python-pptx` 自动创建 PowerPoint 文件。

### 通俗解释：python-pptx

`python-pptx` 是一个用于创建和修改 PowerPoint 文件的 Python 库。

- 文字位置。

### PPT 页面结构

## 4.19 使用标准化函数补齐缺失内容

# 五、项目的一步步迭代历程

## 阶段一：建立最基本的自动化目标

项目最初需要解决的问题是：

> 如何将一篇技术论文自动转化为一份结构清晰的三页简报。

因此首先确定了基础流程：

```text
读取论文
    ↓
调用大模型总结
    ↓
生成结构化内容
    ↓
生成 PowerPoint
```

初始目标主要围绕：

- 核心技术愿景；
- Hardware、Algorithm、Application 三层结构；
- 近期和远期技术趋势；
- 自动化 PPT 输出。

---

## 阶段二：实现文档读取

项目首先实现 PDF 和 TXT 读取功能。

在实现过程中需要处理：

- 文件路径错误；
- PDF 页面提取失败；
- 中文 TXT 编码不同；
- 扫描 PDF 没有可读取文字。

解决方案包括：

- 检查文件是否存在；
- 对每个 PDF 页面单独捕获异常；
- TXT 依次尝试 UTF-8 和 GBK；
- 无文本时提示用户进行 OCR。

实现效果是程序能够较稳定地获得后续分析需要的论文正文。

---

## 阶段三：加入文本清洗

直接从 PDF 中提取的文字通常存在大量断行和空格。

因此加入文本清洗功能：

- 统一换行符；
- 合并普通正文断行；
- 清除多余空白；
- 保留章节和段落边界。

实现效果是提交给模型的文本更接近正常论文段落，降低了排版噪声对模型理解的影响。

---

## 阶段四：解决长论文无法一次处理的问题

整篇论文可能超过模型一次请求能够处理的范围。

因此加入 Chunking：

```text
整篇论文
    ↓
章节识别
    ↓
段落切分
    ↓
句子切分
    ↓
生成多个文本块
```

同时加入 Overlap，让相邻文本块保留部分重复内容。

实现效果是长论文可以通过多次较小请求完成分析，同时保持一定的上下文连续性。

---

## 阶段五：从直接总结改为先提取中间信息

直接让每个文本块生成最终简报，容易出现：

- 每段采用不同表达方式；
- 重要数字被遗漏；
- 只保留名词；
- 不同章节中的重复内容难以合并。

因此引入 NIL：

```text
Named Terms
Important Actions
Literals/Data
Candidate Insights
```

实现效果是每个文本块先形成统一格式的读书笔记，为后续合并和最终写作提供稳定输入。

---

## 阶段六：增加来源追踪

论文被切成多个文本块后，需要知道每条结果来自哪里。

因此给每个 Chunk 分配：

```text
chunk_id
section_title
```

合并后的结果继续保留：

```text
source_chunk_ids
```

实现效果是最终结论仍然能够对应到原始文本块，降低了信息在多次处理后完全失去来源的问题。

---

## 阶段七：从固定关键词改为语义提取

项目需要关注 1000×、Intelligence per Joule 和三大抽象层级，但不能直接将这些词写成所有论文的固定答案。

因此在 Prompt 中加入规则：

```text
关键词只用于定位
不能将关键词直接当作答案
所有结论必须来自论文语义
```

实现效果是模型需要结合上下文判断一个词在论文中的真实含义，而不是看到关键词就机械复制。

---

## 阶段八：加入 NIL 合并

每个 Chunk 都会生成独立 NIL，其中可能存在大量重复信息。

因此增加全局合并阶段：

```text
多个局部 NIL
    ↓
术语去重
    ↓
动作关系合并
    ↓
数据合并
    ↓
高置信度结论
```

实现效果是分散在不同章节中的信息能够被统一整理，为最终简报生成提供一个全局视角。

---

## 阶段九：重新引入原文证据

只依赖合并后的 NIL 可能造成细节损失。

因此加入相关原文选择：

1. 根据关键词为 Chunk 打分；
2. 选择最相关的几个 Chunk；
3. 将这些 Chunk 与 merged NIL 一起交给最终模型；
4. 出现冲突时以原文为准。

实现效果是最终结果既使用了整理后的全局信息，又保留了关键原文依据。

---

## 阶段十：固定最终 JSON 结构

为了让 PPT 生成器稳定工作，项目为最终结果设计了固定 JSON Schema。

结构中明确规定：

- 论文标题；
- 核心愿景；
- 三个抽象层级；
- 两个时间阶段；
- 一句话摘要。

实现效果是模型生成内容与 PPT 页面结构能够一一对应。

---

## 阶段十一：增加 JSON 校验和修复

大模型即使收到严格格式要求，仍可能生成不合法 JSON。

常见问题包括：

- 多余逗号；
- 缺少引号；
- 缺少括号；
- 添加 Markdown 标记。

因此加入两层处理：

```text
严格 JSON 解析
    ↓
失败后调用 JSON Repair
```

实现效果是轻微格式错误可以自动恢复，不需要重新执行全部论文分析。

---

## 阶段十二：增加多模型平台支持

项目最初需要调用大语言模型，因此将接口建立在 OpenAI Python SDK 上。

为了兼容不同平台，进一步加入：

```text
--base-url
--model
--api-key
```

实现效果是程序既可以连接 OpenAI，也可以连接 DeepSeek 等兼容平台。

---

## 阶段十三：加强 API Key 安全

在使用模型 API 时，需要避免密钥泄露。

因此逐步加入：

- 环境变量读取；
- 终端隐藏输入；
- 不将 Key 保存到文件；
- 错误日志自动脱敏；
- README 安全提醒。

实现效果是用户不需要将真实 Key 写入代码，也降低了上传 GitHub和截图时泄露的风险。

---

## 阶段十四：自动生成三页 PPT

完成结构化 JSON 后，项目加入 `python-pptx` 排版功能。

页面逐步确定为：

```text
第 1 页：标题与核心愿景
第 2 页：三大抽象层级
第 3 页：近期到远期时间轴
```

同时加入：

- 16:9 页面比例；
- 统一蓝色主题；
- 圆角卡片；
- 时间轴节点；
- 自动换行；
- 字数和条目数量限制。

实现效果是论文分析结果能够直接转换为一份可演示的 PowerPoint 文件。

# 六、这一实现中采用的核心设计思想

## 6.1 分阶段处理

程序没有使用一个 Prompt 完成所有工作，而是拆成：

```text
局部提取
全局合并
原文复核
最终生成
```

每一阶段只负责一个相对明确的任务。

这种方式降低了单个 Prompt 的复杂度。

---

## 6.2 类似 Map-Reduce 的处理方式

整个 NIL 流程类似 Map-Reduce：

```text
Map：逐个 Chunk 提取 NIL
Reduce：合并全部 NIL
```

### 通俗解释：Map-Reduce

Map-Reduce 是一种将大任务拆成多个小任务，再统一汇总结果的方法。

例如：

```text
10 个人分别阅读 10 个章节
    ↓
每个人提交一份笔记
    ↓
由一个人整理所有笔记
```

项目中的 Chunk 提取相当于 Map，NIL 合并相当于 Reduce。

---

## 6.3 结构化输出优先

项目尽量让模型输出 JSON，而不是普通自然语言。

原因是自然语言适合人阅读，但结构化 JSON 更适合程序继续处理。

```text
自然语言总结
    → 人容易阅读，程序难以稳定拆分

JSON
    → 字段明确，程序可以直接生成 PPT
```

---

## 6.4 内容生成与视觉呈现分离

模型负责生成：

```text
标题
作用
关键特征
技术例子
时间趋势
```

PPT 函数负责决定：

```text
这些内容放在哪里
使用什么颜色
使用多大字号
使用什么形状
```

这种方式避免让模型直接决定所有坐标和排版细节。

---

## 6.5 防御式编程

项目在多个阶段主动检查异常情况，例如：

- 文件是否存在；
- 参数范围是否合法；
- PDF 是否有文字；
- API Key 是否为空；
- 模型是否返回内容；
- JSON 是否完整；
- 层级和时间轴数量是否符合要求。

### 通俗解释：防御式编程

防御式编程是指程序不假设所有输入都正确，而是提前检查可能出现的问题。

它类似于在正式生产前设置多道检查关卡，尽量在问题刚出现时就给出明确提示。

---

# 七、项目解决的主要问题

| 原始问题 | 采用的方法 | 实现效果 |
|---|---|---|
| 长论文无法一次提交给模型 | Chunking 和 Overlap | 将论文拆成多个可处理文本块 |
| PDF 文本混乱 | 文本清洗 | 减少断行和空白对理解的影响 |
| 文本切分后丢失章节背景 | 章节识别与 `section_title` | 保留文本块所属章节 |
| 模型只提取关键词 | NIL 中间表示 | 同时保留术语、关系和数据 |
| 不同文本块信息重复 | NIL Merge | 合并重复信息 |
| 合并后失去原文依据 | `source_chunk_ids` | 保留来源文本块 |
| 多次总结可能失真 | 重新选择相关原文 | 最终生成时再次参考论文 |
| 模型输出结构不稳定 | JSON Schema | 固定字段和层级 |
| JSON 存在轻微格式错误 | JSON Repair | 自动修复括号、逗号和引号 |
| 更换模型平台困难 | OpenAI-compatible API | 可切换 OpenAI、DeepSeek 等平台 |
| API Key 容易泄露 | 环境变量、隐藏输入和脱敏 | 降低密钥泄露风险 |
| PPT 需要人工复制内容 | `python-pptx` | 自动生成三页简报 |
| 模型内容太多导致排版拥挤 | 字数和条目限制 | 控制每页展示的信息量 |
| 字段缺失导致生成失败 | 数据标准化和默认值 | 保证基本页面结构存在 |

# 九、当前实现的边界

## 9.1 主要逻辑集中在单个 Python 文件

文档读取、模型调用、JSON 处理和 PPT 排版都集中在 `generate_briefing.py`。

优点是查看方便、运行简单；但随着功能继续增加，文件会逐渐变长，维护难度也会提高。

---

## 9.2 NIL 使用一次性全局合并

所有 Chunk 的 NIL 会一次性放入一个 Merge Prompt。

当论文很长、Chunk 较多或单个 NIL 内容较丰富时，合并请求可能变得很长。

可能出现：

- 输入 token 较多；
- 输出 JSON 较长；
- 调用成本增加；
- 模型输出达到长度限制。

### 通俗解释：上下文长度

上下文长度是模型一次请求能够接收和生成的文字总量上限。

可以把它理解为模型一次能够同时放在“工作台”上的材料数量。

---

## 9.3 没有保存中间结果

程序运行过程中产生的：

- 各 Chunk 的 NIL；
- merged NIL；
- 最终 JSON；

主要保存在内存中，没有在每个阶段自动写入文件。

如果程序在后期失败，需要重新执行前面的模型请求。

---

## 9.4 没有断点续跑

程序没有记录已经完成到哪个 Chunk。

运行中断后，再次启动通常会从第一个 Chunk 重新开始。

---

## 9.5 JSON Repair 主要处理语法问题

JSON Repair 适合处理：

- 逗号错误；
- 括号错误；
- 引号错误；
- Markdown 包裹。

如果模型输出本身被长度限制截断，后半部分内容根本没有生成，仅靠格式修复无法可靠恢复缺失的信息。

---

## 9.6 相关原文选择采用关键词打分

`_select_context()` 使用预设关键词进行简单打分。

这种方法实现成本较低、速度较快，但不能完全理解所有同义表达。

例如论文没有使用 `efficiency`，而是使用意义相近但表达不同的词语时，可能无法获得较高分数。

---

## 9.7 PPT 文本裁剪主要依据字符数量

程序通过最大字符数限制标题和正文。

这种方法简单有效，但可能在语义没有完整结束时进行截断。

例如一条长句可能被截成前半部分，再添加省略号。

# 十一、项目价值总结

本项目完成了从技术论文到 PowerPoint 简报的自动化转换。

它将一个复杂任务拆分为：

```text
文档处理
文本分块
局部信息提取
全局信息合并
原文复核
结构化生成
格式修复
PPT 排版
```

项目的核心价值不只是调用大语言模型生成总结，而是通过：

- Chunking；
- NIL 中间表示；
- 来源追踪；
- 固定 JSON Schema；
- JSON Repair；
- OpenAI-compatible API；
- 自动化 PPT 排版；

将大模型输出组织为一条可以由程序继续处理的完整流程。

最终实现的是一个能够：

> 读取长篇技术论文、提取跨层级关键信息、整理近期与远期趋势，并自动输出三页技术简报的 Python 工具。
